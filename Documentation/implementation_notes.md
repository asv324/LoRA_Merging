## 2026-03-31 - Track A.2 Synthetic Rotation Recovery

Change: Added `scripts/synthetic_exp1_rotation_recovery.py` and a focused regression check in `tests/test_synthetic_exp1.py`.

Why: Track A.2 requires a reproducible synthetic experiment that measures whether GPA can recover known rotations, consensus structure, and alignment quality across the sweep defined in the implementation plan.

Reasoning: The script stores full per-trial data as requested and also computes per-configuration summaries to make downstream plotting easier. The implementation explicitly resolves GPA's global rotation ambiguity before scoring rotation and consensus recovery, because the observable quantity in `A_i = Q_i A* + sigma E_i` is the aligner `Q_i.T` up to one shared orthogonal transform. The added test uses a noiseless small-scale case to catch mistakes in metric computation before running the full sweep.

Follow-up change: Corrected the synthetic noise scaling in `scripts/synthetic_exp1_rotation_recovery.py` so `E_i` uses the same entry scale as `A*`, and added a regression test in `tests/test_synthetic_exp1.py`.

Why: The first `synthetic_exp1.json` run failed the success criterion because the implementation sampled `E_i` with unit variance while `A*` used variance `1/d`. That made the effective perturbation magnitude scale like `sigma * sqrt(d)`, so with `d = 1536` even `sigma = 0.1` produced noise substantially larger than the signal.

Reasoning: Using the same `1/sqrt(d)` entry scale for both `A*` and `E_i` makes `sigma` dimension-independent and aligns the experiment with the intended interpretation of the parameter sweep. This keeps the synthetic setup suitable for evaluating whether GPA degrades gracefully as controlled noise increases.

Follow-up change: Added `scripts/plot_synthetic_exp1_heatmaps.py` to render the `A.2` heatmaps from `results/synthetic_exp1.json`.

Why: The revised implementation plan calls for heatmaps of rotation recovery error versus `(sigma, N)` at each rank so the synthetic validation can be used directly as dissertation figures.

Reasoning: Keeping plotting separate from the experiment runner makes the figures reproducible from saved JSON without rerunning the full sweep, and also allows the same plotting utility to be reused with other summary metrics if needed.

## 2026-03-31 - Track A.4 Non-Orthogonal Robustness

Change: Added `scripts/synthetic_exp3_nonorthogonal.py` and a focused regression test in `tests/test_synthetic_exp3.py`.

Why: Track A.4 is the main synthetic stress test for the project because real LoRA adapters are not exact orthogonal rotations of a shared matrix. The experiment needs to quantify how GPA behaves as the orthogonality assumption is progressively violated.

## 2026-03-31 - Track A.5 Structured LoRA-Like Ground Truth

Change: Added `scripts/synthetic_exp4_structured.py` and a focused regression test in `tests/test_synthetic_exp4.py`.

Why: Track A.5 is meant to test the most realistic synthetic setting so far by combining a structured shared matrix with task-specific directions, rather than assuming the adapters differ only by clean rotations and noise.

Reasoning: The experiment builds `A*` with geometrically decaying singular values to mimic trained LoRA factors, injects adapter-specific rank-1 perturbations before rotation, and then measures whether the GPA consensus preserves the shared row-space using principal angles. The scalar subspace-overlap summary provides a compact table-friendly metric, while the principal-angle statistics make it easier to interpret how much shared structure is retained as perturbation strength grows.

Follow-up change: Corrected `scripts/synthetic_exp4_structured.py` so the overlap metric is computed on the dominant shared subspace and the residual uses GPA-aligned matrices; updated `tests/test_synthetic_exp4.py` accordingly.

Why: The first `synthetic_exp4_structured.json` run reported about `0.5` overlap even at zero task-specific perturbation, which indicated the metric was comparing the full rank row-space rather than the dominant shared subspace that the plan actually cares about. The same implementation also measured residual against the raw observed matrices instead of the GPA-aligned matrices.

Reasoning: The revised experiment now selects the smallest right-singular subspace of `A*` that explains the configured energy threshold (`0.9` by default) and compares the consensus against that dominant subspace via principal angles. This aligns the metric with the stated goal of recovering the dominant shared structure while leaving tail directions free to absorb noise and task-specific variation. Using the aligned matrices for the residual restores consistency with the GPA objective being optimized.

## 2026-04-05 - Week 2 Step 2.1 Task Arithmetic Baseline

Change: Added `scripts/merge_task_arithmetic.py` and `tests/test_merge_task_arithmetic.py`.

Why: Week 2 Step 2.1 requires a correct Task Arithmetic baseline before implementing the more advanced baselines and the GPA pipeline on real adapters.

Reasoning: For LoRA adapters, treating the saved `A` and `B` factors as if they were ordinary full-model task vectors is not mathematically correct, because separately summing `A` and `B` introduces cross-terms that do not equal the intended weighted sum of adapter deltas. The new baseline therefore merges in effective delta-weight space, where each adapter contributes `(alpha / r) * B @ A`, and stores the exact weighted sum `lambda * sum_i DeltaW_i` using factor concatenation with merged scaling fixed to `1`. The script also strips `modules_to_save` from the merged adapter config so task-specific classification heads are not mixed across GLUE tasks, while still copying tokenizer assets so the merged adapter can be evaluated with the existing standalone evaluation tooling.

## 2026-04-05 - Week 2 Step 2.2 TIES Baseline

Change: Added `scripts/merge_ties.py` and `tests/test_merge_ties.py`.

Why: Week 2 Step 2.2 requires the main raw-adapter baseline that GPA+TIES will be compared against, so it needs to reflect the standard LoRA TIES procedure rather than a custom variant.

Reasoning: The implementation applies trimming, majority-sign election, and disjoint merge independently to the raw `lora_A` and `lora_B` tensors. This matches the revised implementation plan's instruction to operate on the raw, unaligned LoRA weight matrices and avoids accidentally giving the baseline any alignment advantage. Per-adapter LoRA runtime scaling is folded into the factor weights as `sqrt(alpha / r)` on both factors, and those weights are now applied before sign election so the elected sign reflects the scaled contribution when adapters have different LoRA configurations. The script then applies the post-merge `lambda` sweep by scaling the merged `lora_A` tensors, strips `modules_to_save` to avoid mixing task-specific classifier heads across GLUE tasks, and reuses the standalone evaluation path from Week 1.

Follow-up change: Clarified the implementation and metadata as a factor-space TIES baseline rather than exact full-`DeltaW` TIES.

Why: A review of the Week 2 baseline highlighted that TIES on `A` and `B` factors is a pragmatic LoRA-compatible approximation, not the same algorithm as applying TIES directly to reconstructed full deltas `DeltaW = (alpha / r) * B @ A`.

Reasoning: Keeping the merge in factor space preserves the ability to save and evaluate the result as a standard LoRA adapter, which the current Week 2 pipeline depends on. The trade-off is that this baseline still inherits the factor-separability limitation discussed in the KnOTS literature, so the code now labels that limitation explicitly in the saved metadata and documentation to keep the dissertation methodology honest about what is being compared.

## 2026-04-05 - Week 2 Step 2.3 DARE Baseline

Change: Added `scripts/merge_dare.py` and `tests/test_merge_dare.py`.

Why: Week 2 Step 2.3 requires the sparsification baseline that drops redundant adapter parameters before merging, so both the DARE+Task Arithmetic and DARE+TIES comparisons can be run from the same trained adapters.

Reasoning: The implementation applies DARE directly to the stored LoRA factors by randomly zeroing each parameter with probability `p` and rescaling the survivors by `1 / (1 - p)`, which preserves the expected update magnitude of each sparsified tensor. The script then feeds the sparsified adapters into either the exact delta-space Task Arithmetic baseline from Step 2.1 or the factor-space TIES baseline from Step 2.2, keeping the downstream merge behavior consistent with the earlier baselines rather than introducing a third merge variant. A reproducible seed is recorded so the random masks can be regenerated later if the dissertation needs exact reruns of the reported baseline outputs.

Follow-up change: Updated the default DARE sweep to include `p = 0.0`.

Why: Including a no-drop control run makes the default results JSON self-contained, so the DARE experiment matrix can be interpreted without separately joining it against the standalone non-DARE baselines.

Reasoning: The underlying implementation already handled `p = 0` correctly as an identity sparsification step, so changing the default sweep was the smallest way to add the control condition without complicating the rest of the Week 2 and Week 3 experiment pipeline.

## 2026-04-05 - Week 2 Step 2.4 LR-KnOTS Baseline

Change: Added `scripts/merge_lr_knots.py` and `tests/test_merge_lr_knots.py`.

Why: Week 2 Step 2.4 requires the main SVD-based alignment baseline against which the later GPA alignment method will be compared.

Reasoning: The revised plan phrases LR-KnOTS in terms of stacking the `A` factors, running one SVD, merging the per-adapter components with TIES, and reconstructing a merged adapter. The implementation follows that factor-space adaptation by stacking the raw `lora_A` matrices for each module, decomposing the stack into a shared right-singular basis plus per-adapter latent blocks, applying TIES to those latent blocks, and reconstructing `A_merged` before averaging the raw `lora_B` factors. This keeps the merged result loadable as a standard LoRA adapter and makes the comparison to GPA+TIES operationally fair within the project pipeline, while the saved metadata explicitly notes that this is a LoRA-friendly factor-space approximation of the full delta-space KnOTS method from the paper.

## 2026-04-05 - Week 2 Step 2.6 GPA+TIES Factor-Space Merging

Change: Added `scripts/merge_gpa_ties.py` and `tests/test_merge_gpa_ties.py`.

Why: Week 2 Step 2.6 is the primary proposed merge path in the revised plan, combining the real-adapter GPA alignment from Step 2.5 with a conflict-aware merge over the aligned low-rank subspaces.

Why: The v2 plan promotes Experiment 12 to a MEDIUM-priority Week 4 ablation because it is the most direct test of claim C2 — if alignment is the binding constraint in factor-space merging, applying it to a merge as unsophisticated as Task Arithmetic should improve TA substantially; if scale imbalance is the binding constraint, alignment alone should change little. The methodology asks for two main-results-table rows: a plain `GPA-aligned TA` row (which should be numerically identical to unaligned TA up to float noise, because GPA preserves each adapter's effective delta) and an `enhanced-GPA-aligned TA` row combining the headline enhancement ingredients (directional dGPA alignment + inverse-norm `B` weighting) with a TA-style sum rather than a TIES merge. No existing script in the repository could apply dGPA alignment and then merge in TA style, because `merge_task_arithmetic.py` skips alignment entirely and `merge_gpa_ties.py` only supports TIES on the aligned `A` factors.

Reasoning: The pipeline was split across three files to mirror the Step 4.1 N-ablation division of labour and reuse as much Week 3 infrastructure as possible.

- `merge_ta_aligned.py` reuses `align_adapter_bundles` from `scripts/gpa_align_adapters.py` for the alignment step (so the GPA / dGPA code paths used here are bit-identical to those in `merge_gpa_ties.py` — same solver, same normalisation flag, same diagnostics format). It then builds the merged adapter using the same concatenation trick as `scripts/merge_task_arithmetic.py` so the merged output is a valid LoRA adapter that reproduces the intended weighted sum of aligned deltas exactly: `B_merge = [B̃_1, …, B̃_N]`, `A_merge = [λ · s_i · w_i · Ã_i]` stacked by rank, giving `B_merge @ A_merge = λ Σ_i s_i · w_i · B̃_i Ã_i`. Per-adapter weights `w_i` are computed from the aligned-`B` Frobenius norms: `w_i = N · (‖B̃_i‖_F + ε)^(-α) / Σ_j (‖B̃_j‖_F + ε)^(-α)`, so the weights average to exactly 1 across adapters and `α = 0` reduces to `w_i = 1` for all `i`. This choice keeps the overall merge scale comparable across the two variants even when `α > 0` — if the weights had been left to sum to 1 instead (as in `weighted_B_average` inside `merge_gpa_ties.py`), the enhanced row would have been implicitly down-scaled by `1/N` relative to the plain TA reference, confounding "alignment + weighting" with "lower effective λ". The script defaults `--lambdas 1.0` because the methodology pins λ to the best lambda for raw TA from Experiment 6 (`results/best_hparams.json` records `task_arithmetic.hyperparameters.lambda = 1.0`). The default results path is `results/ablation_ta_aligned/ta_aligned_results.json`, but the bash wrapper overrides this on a per-variant basis so the two variants land in separate files (`gpa_aligned_ta.json`, `enhanced_gpa_aligned_ta.json`) that the aggregator can load independently.
- `analyze_ablation_ta_aligned.py` loads each variant file, extracts the primary metric per task using `scripts.data.get_task_config` (the same metric-resolution function used in every other Week 3/4 aggregator so the metric name — `accuracy` for SST-2/MNLI/QNLI/RTE, `matthews_correlation` for CoLA — stays consistent), computes the five-task average, and then emits the delta versus the unaligned Task Arithmetic row loaded from `results/best_hparams.json.methods.task_arithmetic`. The choice of reference matters: Experiment 12's whole point is to compare the two new rows against the unaligned TA headline number, not against the headline GPA+TIES or the enhanced TIES row, because C2 is about alignment feeding into an otherwise-fixed merge style. The summary includes the per-task delta and a `positive_task_count` so the Discussion-chapter C2 section can read off whether the TA row moved up or down on each of the five tasks (especially whether the CoLA Matthews correlation stays at exactly `0.0`, matching the CoLA-collapse pattern documented in Week 3). ## 2026-04-21 - Deterministic classifier-head init in `adapter_eval.py`

Change: Added a fixed seed (`DEFAULT_EVAL_SEED = 0`, overridable via the `LORA_MERGE_EVAL_SEED` environment variable) that is applied via `transformers.set_seed()` immediately before every `AutoModelForSequenceClassification.from_pretrained` call in `scripts/adapter_eval.py`. The resolved seed is recorded on the returned metrics dictionary as `classifier_head_init_seed` so downstream JSON files are self-documenting.

Why: The five Week 1 GLUE adapters were trained with `modules_to_save = ["score", "classifier"]`, so each source `adapter_model.safetensors` stores a fully trained `base_model.model.score.weight`. However, every merge script in the repository (`merge_task_arithmetic.py`, `merge_gpa_ties.py`, `merge_lr_knots.py`, `merge_dare.py`, `merge_ta_aligned.py`) writes only `lora_A`/`lora_B` keys into the merged state dict and overwrites `modules_to_save = None`, so the merged adapter does not restore any classifier head. When `PeftModel.from_pretrained` loads the merged adapter on top of `Qwen2.5-1.5B`, `score.weight` is freshly random-initialised by `AutoModelForSequenceClassification.from_pretrained`. Without a fixed seed, the draw varies across eval processes, and the Experiment 12 verification run exposed the practical consequence: plain Task Arithmetic at λ = 1 gave SST-2 = 0.797 in the original Experiment 6 sweep and SST-2 = 0.548 in the fresh Step 4.2 run even though the per-module `merged_delta_frobenius_norm` values are bit-identical (max absolute difference 3.8e-6, max relative 1.8e-7 across all 196 LoRA modules). The SST-2 accuracy trajectory across λ in the original sweep (0.491, 0.471, **0.335**, 0.500, 0.678, 0.638, 0.490, 0.513, 0.491, **0.797**) is the signature of a random binary classifier interacting with a λ-dependent LoRA delta — values below chance and chaotic with respect to λ are inconsistent with a meaningfully-trained head. QNLI, RTE, and CoLA appear constant across every sweep because their random classifiers collapse to a single-class predictor that is insensitive to the LoRA delta; SST-2 is the only task where the binary-balanced validation set is sensitive to where the random hyperplane lands.

Reasoning: A fixed pre-load seed is the minimum-invasive intervention that removes process-to-process classifier-head variance without changing the actual merge semantics. It keeps every existing merge script untouched and does not silently restore the trained head (which would also be valid but would retroactively change the numerical value of every Week 3 / Week 4 result already on disk). Using `transformers.set_seed()` covers `random`, `numpy`, and `torch` simultaneously, matching what the HuggingFace training pipeline assumes. The seed is applied per-`evaluate_adapter` call (i.e. per task) rather than once per process, so that task ordering inside `evaluate_adapter_on_tasks` cannot shift the classifier-head draw for later tasks. `LORA_MERGE_EVAL_SEED` is read at call time, so downstream orchestrator scripts (`run_week3_statistical_significance.sh`, `run_step4_1_ablation_N.sh`, `run_step4_2_ablation_ta_aligned.sh`) can override the seed without code changes if a sensitivity analysis over classifier-head draws is ever needed.

Notes, assumptions, and expected impacts:

- After this change, re-running the same merged adapter through `scripts/adapter_eval.py` will produce identical metrics across processes (up to BitsAndBytes quantisation / fp16 kernel nondeterminism, which is orders of magnitude below the score-head variance the seed is fixing). Two merged adapters that share the same effective ΔW (e.g. plain TA and `gpa_aligned_ta`) will now produce identical classifier outputs by construction.
- Every evaluation result JSON produced before this change (all of `results/hp_sweep_low_storage/`, `results/statistical_significance/`, `results/ablation_N/`, and the Step 4.2 files in `results/ablation_ta_aligned/`) was generated with an unseeded classifier-head draw. The merged-delta numerics in those files are still correct, but the SST-2 accuracy values in particular carry the process-to-process random-head noise documented above. Re-running with the new seed is expected to change SST-2 numbers visibly; QNLI, RTE, and CoLA should remain at their majority-class or degenerate values because they are already saturated at the all-one-class prediction baseline under any random head.
- This change does not affect the Step 3.2 best-hyperparameter selection logic or the Step 3.3 seed-variance comparisons *relative to each other within a single process*, because those runs evaluated every method under the same process RNG state. It does, however, affect the absolute values reported, and — most importantly — makes cross-process comparisons (Experiment 6 headline numbers versus fresh Experiment 12 reruns, or fresh Experiment 11 N-ablation cells versus the reused Experiment 6 N = 5 row) internally consistent again.
- The long-term fix is to restore the trained classifier head in the merged adapter, which removes the random draw entirely and is also the semantically correct behaviour for multi-task merging on GLUE. That change is left for a follow-up because it also requires deciding which task's head to carry (or how to merge heads across tasks) and has downstream implications for the dissertation's discussion of QNLI / RTE / CoLA saturation as a structural rather than seed-level effect.

## 2026-04-21 - Per-task classifier-head preservation across merge → eval round-trip

Change: Merged adapters now store the trained classifier head of every source task alongside the LoRA factors, and `scripts/adapter_eval.py` restores the head matching the evaluated task before running `Trainer.predict`. Concretely:

- Added a shared `copy_classifier_heads(adapter_bundles, output_dir)` helper in `scripts/merge_task_arithmetic.py`, plus four module-level constants (`CLASSIFIER_HEAD_SUBDIR = "classifier_heads"`, `CLASSIFIER_HEAD_SOURCE_KEY = "base_model.model.score.weight"`, `CLASSIFIER_HEAD_OUTPUT_KEY = "score.weight"`, `CLASSIFIER_HEAD_MANIFEST_FILENAME = "classifier_heads_manifest.json"`). The helper writes `<merged_adapter>/classifier_heads/<task>.safetensors` (keyed by `score.weight`) for every bundle that has a trained head, and emits a `classifier_heads_manifest.json` listing the per-task shape, dtype, source adapter directory, and relative path. Bundles without a `score.weight` tensor (hypothetical LM-only adapters) are skipped silently so the helper is safe to call unconditionally.
- Wired the helper into the `save_merged_adapter` / `write_merged_adapter` call sites of every merge script in the repository (`merge_task_arithmetic.py`, `merge_ta_aligned.py`, `merge_gpa_ties.py`, `merge_ties.py`, `merge_dare.py`, `merge_lr_knots.py`) via a new optional `adapter_bundles` argument that defaults to `None`. Passing `None` preserves the pre-change behaviour so any external caller that imports these functions is not forced to update. The DARE path passes `sparsified_bundles` (rather than the raw bundles) because `apply_dare_to_adapter_bundles` preserves non-LoRA keys via `tensor.clone()`, so each sparsified bundle still carries the original trained head and is the correct object to keep the saved head consistent with the `adapter_dir` field recorded in the merged metadata.
- Added `_restore_classifier_head(model, adapter_path, task, num_labels)` to `scripts/adapter_eval.py`. After `PeftModel.from_pretrained` loads the merged adapter, this helper reads `<adapter_path>/classifier_heads/<task>.safetensors`, validates the shape against `(num_labels, hidden_size)` of the loaded `model.base_model.model.score.weight`, and copies the tensor onto the score module. If `BitsAndBytesConfig(load_in_4bit=True)` caused `score` to be replaced by `bitsandbytes.nn.Linear4bit`, the helper swaps in a fresh full-precision `torch.nn.Linear` rather than quantising the trained head to NF4 (the head is only `(2 or 3) × 1536` parameters, so full precision costs a trivial 6-18 KB per adapter and avoids NF4 quantisation error on the exact values we are trying to preserve). If no head file exists (legacy pre-2026-04-21 merged adapters or LM-only merges), the helper returns without mutating the model, and the seeded-random-init path from the previous fix keeps the evaluation deterministic. The diagnostic dict it returns (`restored`, `reason`, `path`, `shape`, `saved_dtype`, `runtime_dtype`, `replaced_quantized_score`) is attached to the per-task metrics JSON under `classifier_head_restoration` so every downstream analysis can check which head was loaded and whether bnb replacement occurred.

Why: The 2026-04-21 deterministic seed fix turned the process-to-process variance of the random classifier head into a reproducible number, but it did not recover the *semantic* signal. Every merged adapter in the repository still evaluates against a random hyperplane through `hidden_dim`-space — which is why SST-2 accuracy at `λ = 1` for plain Task Arithmetic bounces between 0.3349 and 0.7970 across λ values even though the per-module `merged_delta_frobenius_norm` values are bit-identical across merge variants, and why QNLI/RTE/CoLA have been saturated at majority-class prediction across every Week 3 and Week 4 run. The Week 1 adapters were trained with `modules_to_save = ["score", "classifier"]` and therefore carry a fully trained `score.weight` in their `adapter_model.safetensors`, so there is no need to retrain — the head already exists; it was simply being discarded during the merge. This change restores it.

Reasoning: The central design decision is that GLUE tasks have *different* output-label counts (MNLI has 3, SST-2/QNLI/CoLA/RTE have 2) and therefore *different* `score.weight` shapes (`[3, 1536]` versus `[2, 1536]`), which means there is no single "merged classifier head" that can serve all evaluations. Four design alternatives were considered:

1. Pick one task's head arbitrarily (e.g. always use SST-2's head): rejected because it is wrong for MNLI (3-way), and evaluating MNLI under a 2-way head is impossible.
2. Average heads of matching shape: rejected because averaging classifier planes across tasks is theoretically meaningless (the labels are different concepts) and would not help MNLI anyway.
3. Store a single merged head that routes by task: rejected as over-engineered for the actual use case.
4. **Store every source task's head and restore the matching one at eval time**: chosen. This is the only option that is both semantically correct (each task evaluates under its own trained head) and implementable without changing the saved merge-delta semantics (`ΔW = B @ A` is unchanged; we only add extra files next to the LoRA factors).

The on-disk layout (`classifier_heads/<task>.safetensors` with a single `score.weight` key, plus a sibling `classifier_heads_manifest.json`) is chosen for three reasons: (a) PEFT does not consume these files, so storing them outside `adapter_model.safetensors` keeps the merged adapter a valid stand-alone LoRA adapter that any `PeftModel.from_pretrained` can load without custom code; (b) one small safetensors file per task keeps the tensors self-describing and discoverable without parsing compound keys; (c) the manifest gives downstream analysis scripts a single JSON to consult for what tasks were saved, their shapes, and their source adapter directories without scanning the filesystem. The source adapters use the key `base_model.model.score.weight`, but the saved head drops the `base_model.model.` prefix to `score.weight` so the key matches the attribute path on a freshly-loaded `AutoModelForSequenceClassification` rather than the PEFT-wrapped path — this makes the files directly loadable into a raw HF model in any context (tests, inspection scripts) without requiring the PEFT wrapper.

On the eval side, the key subtlety is `BitsAndBytesConfig(load_in_4bit=True)`: if bnb replaces `score` with `Linear4bit`, naive `target_module.weight.copy_(head_weight)` fails because `Params4bit` does not accept arbitrary fp32 assignments. The chosen workaround — replacing the quantised score with a fresh fp16 `nn.Linear` — keeps the score head in full (fp16) precision rather than NF4, which is correct: the whole point of restoring the trained head is to preserve its exact classifier plane, and NF4 quantisation of a `(num_labels, 1536)` layer would introduce quantisation error on the very values we are trying to recover. The memory cost is negligible (`3 × 1536 × 2 bytes ≈ 9 KB` for MNLI, `2 × 1536 × 2 bytes ≈ 6 KB` for the others), and the matmul against fp16 transformer activations works unchanged because `nn.Linear` with fp16 weights is a first-class PyTorch path.

A fallback path (random init seeded per the 2026-04-21 fix) is retained for adapters that do not have saved heads: if `classifier_heads/<task>.safetensors` is missing, `_restore_classifier_head` returns `{"restored": False, "reason": "no_classifier_head_file"}` without touching the model, and the deterministic seed keeps the eval reproducible. This matters because pre-change merged artifacts (all of `results/hp_sweep_low_storage/`, `results/statistical_significance/seed_*/*.json`, `results/ablation_N/`, and the initial `results/ablation_ta_aligned/` files) do not yet carry these heads, and we want eval code that remains backwards-compatible so researchers can load those artifacts without having to re-merge.

Unit coverage:

- `tests/test_merge_task_arithmetic.py::CopyClassifierHeadsTests::test_round_trip_preserves_per_task_heads` constructs three synthetic bundles with differently-shaped heads (mirroring the SST-2 / MNLI / RTE shape pattern from the real adapters), calls `copy_classifier_heads`, and asserts every saved file contains exactly `{"score.weight": <tensor>}` with the original values and that the manifest matches the in-memory return value. ## 2026-04-21 - Fixed `ADAPTERS_SEED_TEMPLATE` parsing bug in `scripts/run_week3_statistical_significance.sh`

Change: Replaced the single-line `${VAR:-default}` assignment

```bash
ADAPTERS_SEED_TEMPLATE="${ADAPTERS_SEED_TEMPLATE:-${PROJECT_ROOT}/adapters/seed_{seed}}"
```

with an explicit `if [[ -z "${ADAPTERS_SEED_TEMPLATE:-}" ]]; then ... fi` guard (and a comment explaining the hazard). The default value (`${PROJECT_ROOT}/adapters/seed_{seed}`) is unchanged, and the `{seed}` placeholder substitution at `resolve_adapters_dir()` (currently line ~149) still works identically.

Why: The old line had a latent bash-parser bug that silently corrupted any user-supplied value of `ADAPTERS_SEED_TEMPLATE`. In this line the default ends with `seed_{seed}}`: the `}` right after `{seed` closed the outer `${...}` and the trailing `}` was re-appended as a literal byte to the expansion result. This was discovered when Step B of the Week 4 rerun plan tried to point the driver at an existing adapter tree via `ADAPTERS_SEED_TEMPLATE=adapters` and the banner printed `Adapters seed template: adapters}`, with the subsequent `Missing seed-specific adapters under adapters}.` error as the symptom.
